# Deliberative Layer — Design Session Summary

**Date:** 2026-05-16  
**Status:** Reference summary  
**Covers:** The full design discussion of 2026-05-16, spanning argument vocabulary, multi-Claude
deliberation, DevTown integration, SDLC framing, and infrastructure tiers

---

## The Central Idea

Qhorus already handles the *commitment* layer: who promised what to whom, when, and what happened.
What it does not yet handle is the *deliberative* layer — the structured back-and-forth when agents
are not executing instructions but **reasoning together toward a decision**.

The argument graph is the designed answer. Its primary purpose is accountability and explainability:
a human-readable, human-verifiable record of *why* a conclusion was reached — not just that it was
reached and who committed to it.

```
The normative ledger answers:  WHAT was decided | WHO committed to it
The argument graph answers:    WHY — the reasoning path that produced the conclusion
```

---

## Foundation First: This Is a Qhorus Primitive

The argument vocabulary and graph are a **Qhorus foundation layer** — a general-purpose
deliberation primitive available to any application built on Qhorus or CaseHub, in the same way
the normative layer (COMMAND, DECLINE, DONE, FAILURE…) is a general-purpose speech-act primitive.

The vocabulary is not a DevTown feature. It is a building block. DevTown is one application that
uses it. Other applications — any multi-agent system requiring structured deliberation and
accountability — can use it without DevTown. The layers:

```
┌─────────────────────────────────────────────────────────┐
│  APPLICATION LAYER                                      │
│  DevTown (review + trust) | multi-Claude debate skill  │
│  Any other agent system needing structured deliberation │
└─────────────────────────────────────────────────────────┘
                          ↑ uses
┌─────────────────────────────────────────────────────────┐
│  ARGUMENT LAYER  (Qhorus foundation — this design)     │
│  Vocabulary: CLAIM PRESUME SUPPORT REBUT UNDERCUT       │
│              CONCEDE REVISE                             │
│  Graph: extraction, checkpoint, ratification            │
│  Infrastructure: tier 1 → tier 2 → tier 3              │
└─────────────────────────────────────────────────────────┘
                          ↑ sits above
┌─────────────────────────────────────────────────────────┐
│  NORMATIVE / COMMITMENT LAYER  (existing)               │
│  COMMAND DECLINE HANDOFF DONE FAILURE EVENT             │
│  CommitmentStore, MessageLedgerEntry, obligations       │
└─────────────────────────────────────────────────────────┘
                          ↑ sits above
┌─────────────────────────────────────────────────────────┐
│  CHANNEL / SPEECH ACT LAYER  (existing)                 │
│  Typed messages, ChannelGateway, instance registry      │
└─────────────────────────────────────────────────────────┘
```

The parallel to the normative layer is exact: the normative layer did not teach LLMs what a
COMMAND is — they already understood speech acts. It gave them shared, standardised labels so the
infrastructure could derive the obligation lifecycle. The argument layer does the same for logical
moves: LLMs already know what a rebuttal is. The vocabulary gives them shared labels so the
infrastructure can derive the reasoning trail.

## The Argument Vocabulary

Seven moves cover the logical structure of deliberative conversation:

| Move | What it captures |
|---|---|
| **CLAIM** | A position asserted with stated reasoning |
| **PRESUME** | An assumption the argument depends on — stated, explicitly inviting challenge |
| **SUPPORT** | Evidence or reasoning backing a CLAIM |
| **REBUT** | Direct contradiction of a specific CLAIM |
| **UNDERCUT** | Challenge to the assumption or evidence behind a CLAIM — not the CLAIM itself |
| **CONCEDE** | Explicit withdrawal of a prior position, with the reason |
| **REVISE** | An updated CLAIM caused by a successful REBUT or UNDERCUT |

The vocabulary is intentionally close to natural language. Agents are not asked to produce formal
logic — only to label the logical move they are making inline, in the same text channel as the
conversation. The graph is extracted from the labelled text at checkpoint moments, not built
separately in real time.

**PRESUME + UNDERCUT is the hinge pair.** Both worked examples in the argument-graphs spec show
this: surfacing a hidden assumption so it can be challenged directly produces a better outcome than
either starting position. The PRESUME move — explicitly stating an assumption and inviting
correction — is the most important move to emphasise in any system prompt.

**Conformity is visible.** A key design property: an agent cannot silently shift position without
registering a CONCEDE or REVISE move. Conformity under social pressure becomes a recorded act,
not an invisible drift. This addresses the "minority correction asymmetry" identified in the
literature (arXiv 2511.07784).

## Three Infrastructure Tiers

The same vocabulary operates at all three tiers. Infrastructure is the variable, not the moves.

### Tier 1 — Vocabulary only (available now)

The seven moves used in normal conversation text. No tooling, no channel, no storage. The
vocabulary IS the lightweight version — nothing needs to be built to use it.

**The trade-off:** No automatic state tracking. No convergence detection. Human carries the graph
state or uses a scratch document.

### Tier 2 — Lightweight Qhorus channel (engineering track)

An ephemeral `debate/<session-id>` channel that two Claude instances can both post to and read
from. Designed for the terminal use case: working with Claude, wanting to launch a second Claude
for critique, not wanting to mix two large conversations into one context.

**What it provides:** Conversation routing (contexts stay separate); argument graph as a live
navigation aid; consensus detection. **What it does not provide:** Ledger entries, attestation,
provenance, normative layer enforcement.

The channel is discarded at session end. It is a conversation coordination mechanism, not a
governance artefact.

### Tier 3 — Full Qhorus (governance grade)

Each argument move creates a tamper-evident `MessageLedgerEntry`. Ratification is a formal deontic
act — a signed checkpoint carrying the same weight as a DONE in the commitment layer. The argument
graph is a permanent, queryable artefact with full causal chain from first CLAIM to ratified
conclusion.

For contexts where the WHY of a decision has regulatory weight: compliance, security design,
financial system architecture, any decision that may require explanation to an auditor.

### Retroactive Promotion

A tier 1 conversation using the vocabulary can be promoted to a tier 3 record at any point — before
a final decision, before implementation, before a design is committed. The argument structure is
already in the text. Extraction is pattern-matching on labelled moves. The governance decision
does not need to be made at the start of a deliberation.

---

## Two Operational Tracks

Independent of tier, the deliberation layer operates at two levels:

**Engineering track:** Helps produce better decisions. No ledger, no provenance, no compliance
requirement. Most use of the vocabulary will be here. Overhead is near-zero; value is immediate.

**Governance track:** Full accountability with ledger, provenance, and ratification as a formal
act. For regulated contexts where the reasoning trail has legal or compliance weight.

Same vocabulary, same moves, different infrastructure underneath. The track is a deployment
decision, not a design constraint.

## Three Parts of the SDLC

The design work has surfaced three distinct SDLC concerns, each answered by different tooling:

### Triage

*Which work deserves attention?*  
The trust model. PR volume is too high to review everything; quality varies; AI-generated
submissions are increasing. Trust scores route high-signal work to the right reviewers and protect
human attention. This is DevTown's core concern — not the work itself, but who and what deserves
review time.

### Review

*Is this work correct?*  
Assessment of a finished or near-finished artifact. Reviewer (human or agent) evaluates, flags
issues, approves or rejects. GasTown's core workflow, extended by DevTown. The output is a verdict.
The record is: what was reviewed, by whom, and what was decided.

**Review is assessment of a finished artifact.**

### Deliberation

*What should we build, and how?*  
Structured argument during the creation of work. Positions taken, assumptions challenged, positions
revised. Deliberation happens primarily before or during implementation, but can also occur inside
review when reviewer and author disagree.

**Deliberation is argument during the creation or critique of work, not assessment of it.**

The SDLC flow:

```
Deliberation (design / implementation phase)
      ↓ work produced
Triage (which PRs deserve attention?)
      ↓ selected for review
Review (is this work correct?)
      ↓ ← deliberation may nest here when reviewer and author disagree
Verdict
```

## Multi-Claude Deliberation

The pattern: instead of the human providing all critical pressure, launch a second Claude with a
specific role — critic, devil's advocate, security skeptic, simplicity advocate, custom framing —
and have it argue with the primary Claude using the argument vocabulary.

The human becomes **overseer rather than reviewer**: watching the argument graph state,
intervening when needed, ratifying when consensus is reached.

**Critique roles:**

| Role | Angle |
|---|---|
| critic | General flaw-finder |
| devil-advocate | Argues against current position regardless of merit |
| security | Attack surfaces, trust boundaries, data risks |
| performance | Load, latency, what breaks at scale |
| simplicity | Is this complexity justified? What can be removed? |
| alternative | Argues for a specific named alternative |
| [custom] | Any description — "argue as a sceptical CTO who has seen this fail before" |

**Consensus** means the argument graph has converged — no outstanding REBUTs or UNDERCUTs.
Not that everyone agrees; that every dispute has reached a terminal state. Open disagreements
surface as UNRESOLVED. The human decides whether to accept the consensus with open items, resolve
them, or run another round.

**PRESUME seeding at launch** is the critique accelerator: if the primary Claude explicitly states
its assumptions as PRESUMEs in the opening CLAIM, the critique Claude can target them directly with
UNDERCUTs rather than inferring assumptions from free text. This produces faster, more precise
critique.

**The `/debate` skill:**

```
/debate --role <role> [--rounds N] [--intensity gentle|standard|adversarial] "<topic>"
```

Packages the primary Claude's current position as a structured brief, launches the critique Claude
seeded with the role brief, opens a debate channel, and runs the argument loop. The human sets the
role and stopping condition at launch and then watches the panel.

**The cross-repo case is a deployment variant, not a separate design.** When the primary and
critique Claudes happen to be working in different codebases, the channel routes between them.
The pattern is identical.

## DevTown Integration

This section records how the foundation layer connects to DevTown specifically. DevTown is an
application on top of the argument layer — not the definition of it.

**The join point:** DevTown's `HumanOversight` trigger. When trust is borderline and the system
cannot confidently route a decision, the human is surfaced. Instead of reviewing directly
(HumanDecision), they launch a critique Claude (human-as-overseer). The debate resolves the
uncertainty.

**Trust phase to critique intensity:**

| DevTown Phase | Debate Intensity |
|---|---|
| Phase 1 — emerging | gentle |
| Phase 2 — active | standard |
| Phase 3 — adaptive | adversarial |

**Argument moves as trust signals:**

| Signal | Trust dimension |
|---|---|
| REBUT/UNDERCUT → primary REVISE | REVIEW_THOROUGHNESS (critique found real issues) |
| REBUT/UNDERCUT → primary CONCEDE (challenge rejected) | FALSE_POSITIVE_RATE (critique flagged non-issues) |
| Critique CONCEDEs out-of-domain | SCOPE_CALIBRATION (critique stayed in its lane) |

**The trust subject for critique Claudes** is the role (durable, accumulates across debates), not
the instance (ephemeral, destroyed after the debate). Issue #24's Bayesian Beta + EigenTrust model
applies to roles the same way it applies to contributors.

**The gap the argument layer fills:** DevTown's normative ledger records what was decided and by
whom. It does not record why. The argument graph is the explainability layer: what assumptions
drove the conclusion, where they were challenged, and what caused the position to change or hold.

## Validation Before Infrastructure

The deliberation-probe skill (`docs/skills/argument-sweep.md`) is a read-only evaluation tool:
it scans an existing conversation and reports how well the vocabulary maps onto what actually
happened. Coverage, explicit/implicit ratio, hinge moves, UNCLASSIFIED moves.

Run this across several conversations before building Phase 2 infrastructure. The vocabulary must
prove it works in natural conversation before the persistence layer is built for it. This follows
the same validation path as the normative layer: conceptual correctness before CommitmentStore.
