## #257 — Separate EVENT telemetry from content; fail-fast for application callers

### Problem

`MessageDispatch.Builder` accepts `.content(String)` for `MessageType.EVENT` without any guard. The content is silently nulled by `MessageObserverDispatcher` before delivery, so application observers receive `null`. No error surfaces at dispatch time — data disappears.

Separately, `LedgerWriteService.populateTelemetry()` extracts MCP tool telemetry (`tool_name`, `duration_ms`, `token_count`, `context_refs`, `source_entity`) by parsing `dispatch.content()` as JSON. This conflates the application-visible content field with an internal ledger concern, which is why content cannot be used for application-level EVENT payloads.

### Root cause of the coupling

`message.content` is the application layer. `MessageLedgerEntry.toolName` etc. are the ledger layer. Using the same field (`content`) to carry both is the architectural error. The proper fix is in `casehub-ledger#126`, which will move telemetry capture to a mechanism that does not touch `content`. This issue is the partial step: introduce an explicit `telemetry` field in `MessageDispatch`, route internal telemetry callers to it, and enforce that application callers cannot set `content` on EVENTs.

### Design

**Part A — Add `telemetry(String)` to `MessageDispatch`**

Add a 13th field `String telemetry` to the `MessageDispatch` record, after `deadline`. Field Javadoc:

```java
/** Internal: EVENT-only ledger telemetry payload; parsed by LedgerWriteService; never delivered to observers. */
String telemetry
```

Add a corresponding `telemetry(String v)` setter to `Builder`. No validation on `telemetry` in `build()` — it is internal infrastructure, not consumer API.

`LedgerWriteService.populateTelemetry()` and `ReactiveLedgerWriteService.populateTelemetry()` both change their parameter from `dispatch.content()` to `dispatch.telemetry()`. This is the decoupling: telemetry extraction reads from an explicit field, not the application content field.

**Part B — Fail-fast guard in `Builder.build()`**

Add to the `switch` in `build()`:
```java
case EVENT -> {
    if (content != null)
        throw new IllegalArgumentException(
            "EVENT messages must not carry content — use STATUS for content-bearing observe-channel broadcasts.");
}
```

The reference to `casehub-ledger#126` belongs in Javadoc, not the runtime exception. Agents receive `ToolCallException` messages and do not benefit from cross-repo issue references.

**Part C — Update all 8 production EVENT callers**

| Caller | Current | Change |
|--------|---------|--------|
| `ChannelGateway.receiveHumanMessage()` | `.content(telemetryJson)` | → `.telemetry(telemetryJson)` |
| `ChannelGateway.receiveObserverSignal()` | `.content(signal.content())` | Remove — drop the payload (see fanOut note below) |
| `QhorusMcpTools.force_release_channel` | `.content(auditContent)` | Remove — the EVENT record is the audit |
| `QhorusMcpTools.delete_message` | `.content("delete_message: ...")` | Remove |
| `QhorusMcpTools.clear_channel` | `.content("clear_channel: ...")` | Remove |
| `ReactiveQhorusMcpTools.force_release_channel` | `.content(auditContent)` | Remove |
| `ReactiveQhorusMcpTools.delete_message` | `.content("delete_message: ...")` | Remove |
| `ReactiveQhorusMcpTools.clear_channel` | `.content("clear_channel: ...")` | Remove |

**Observer signal fanOut behavioral change:** After dropping `signal.content()` from `receiveObserverSignal()`, `MessageService.dispatch()` will call `channelGateway.fanOut()` with an `OutboundMessage` carrying `content=null`. Previously it carried the signal payload. Any external backend whose `post()` implementation reads `OutboundMessage.content()` for observer signals will see null where it previously had the signal payload. This is intentional: `signal.content()` was already null to `MessageObserver` implementations (nulled by the dispatcher), and `ChannelBackend.post()` for observer signals should not rely on content. Update `OutboundMessage` Javadoc to state that `content` is null for EVENT.

**`MessageObserverDispatcher` nullification**

The ternary `message.messageType == MessageType.EVENT ? null : message.content` becomes dead code for all production paths: after Part B+C, no production EVENT dispatch can carry non-null content. The guard remains in place as defense-in-depth against direct `Message` entity construction that bypasses the `Builder`. Update the Javadoc to say this explicitly — do not reference `casehub-ledger#126` here, which is about telemetry storage decoupling, not about the nullification.

**`MessageReceivedEvent` compact constructor**

The `if (messageType == MessageType.EVENT && content != null) throw` guard remains. Update Javadoc: "content is null for EVENT because the Builder guard prevents non-null content from reaching the dispatcher. Once casehub-ledger#126 fully decouples telemetry capture from content, the architectural decision on whether EVENT should support application content can be made explicitly at that time."

## #258 — ChannelSlugValidator: actionable error for dot-notation names

### Problem

`ChannelSlugValidator.validateSlugPath()` splits on `/`, so `quarkmind.scouting.intel` is one segment. The error repeats the full name in both the "segment" and "full name" positions and does not explain that dots are invalid or suggest any correction.

### Fix

After the existing `!SEGMENT_PATTERN.matcher(segment).matches()` check, branch on whether `segment.contains(".")`. When a dot is found, generate a specific message:

```
Invalid channel name segment 'quarkmind.scouting.intel' in 'quarkmind.scouting.intel'
— dots are not valid segment characters ('/' is the qhorus path separator, not '.').
Suggested alternatives: 'quarkmind-scouting-intel' (hyphen) or 'quarkmind/scouting/intel' (hierarchy).
Segment must match [a-z][a-z0-9]*(-[a-z0-9]+)*.
```

The dot-to-hyphen and dot-to-slash suggestions are computed from the segment string (replace `.` with `-` and `.` with `/` respectively), not hardcoded. For non-dot failures (uppercase, digit-start, trailing hyphen), the existing error format is retained.

### Scope

- `ChannelSlugValidator.validateSlugPath()` — branch on `segment.contains(".")` after pattern mismatch
- Javadoc on `validateSlugPath()` — document that `/` is the path separator and dots are invalid

### Tests

- `quarkmind.scouting.intel` → error contains both `quarkmind-scouting-intel` and `quarkmind/scouting/intel`
- `quarkmind/scouting.intel` (dot in a non-first segment) → first segment passes, second triggers dot-specific error
- `INVALID` (uppercase, non-dot) → existing error, no dot hint
- `123-invalid` (digit-start, non-dot) → existing error format
- `valid/path/name` → passes validation

## #259 — Add `@Any` to `Instance<MessageObserver>` injection

### Problem

Both `MessageService` and `ReactiveMessageService` inject `Instance<MessageObserver>` without `@Any`. CDI's unqualified `Instance<T>` resolves only beans with the `@Default` qualifier. Any `MessageObserver` implementation with an additional qualifier (`@Named`, a custom `@CaseType`, any application qualifier) is silently excluded from `observers.handles()`.

### Fix

```java
// Before
@Inject
Instance<MessageObserver> observers;

// After
@Inject @Any
Instance<MessageObserver> observers;
```

Applied identically in `MessageService` and `ReactiveMessageService`. `InProcessMessageBus` (`@DefaultBean`) retains `@Default` and continues to be discovered. The channel filter (`observer.channels()`) and scope field (`observer.scope()`) are applied after handle resolution — unaffected.

Update `MessageObserver` Javadoc: "Implementations may carry additional CDI qualifiers (e.g. `@Named`, custom application qualifiers) — the dispatcher discovers all implementations regardless of qualifiers."

### Test

A `@QuarkusTest` with a custom-qualified `MessageObserver`. The qualifier annotation must declare:

```java
@Qualifier
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.TYPE, ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER})
```

Without `@Retention(RUNTIME)`, Arc will not see the qualifier at runtime.

Test assertions:
1. After `messageService.dispatch()`, the qualified observer's `onMessage` is called.
2. A qualified observer with a non-empty `channels()` filter receives only matching messages and is excluded from others — confirms that channel filter and qualifier discovery compose correctly.

A separate reactive-path test is not needed: the CDI injection is resolved at augmentation time and shared by both stacks.

## Coherence review

- **`MessageDispatch` canonical constructor break** affects 6 call-sites (excluding the Builder itself and the two helper abstractions). All are mechanical null-appends. `ChannelGateway` and `LedgerWriteServiceTest` are qhorus-internal; no downstream consumer constructs `MessageDispatch` directly — they use the Builder.
- **The Builder `case EVENT` guard** is a consumer-facing break for any caller that sends `EVENT+content`. That breakage is intentional — it makes the previously silent data loss into a compile-time/runtime error.
- **`@Any` on `Instance<MessageObserver>`** is backwards-compatible — existing default-qualified beans continue to be discovered.
- **PLATFORM.md `observe` channel semantics** (EVENT, QUERY, STATUS allowed) are unchanged. EVENT remains a signal type at the SPI boundary pending `casehub-ledger#126`.
